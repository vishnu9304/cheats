    Sessions

    Consul Sessions

    1. Consul provides a session mechanism which can be used to build distributed locks.
    2. A session in Consul represents a contract that has very specific semantics.
    3. When a session is constructed, a node name, a list of health checks, a behavior, a TTL, and a lock-delay may be provided.
    4. The newly constructed session is provided with a named ID that can be used to identify it.
    5. This ID can be used with the KV store to acquire locks.
    6. The session will be invalidated under below situations:
      • Node is deregistered.
      • Any of the health checks are deregistered.
      • Any of the health checks go to the critical state.
      • Session is explicitly destroyed.
      • TTL expires, if applicable.
    7. When a session is invalidated, it is destroyed and can no longer be used.
    8. What happens to the associated locks depends on the behavior specified at creation time.
    9. Consul supports a release and delete behavior.
    10. The release behavior is the default if none is specified.
    11. If the release behavior is being used, any of the locks held in association with the session are released, and the ModifyIndex of the key is incremented.
    12. Alternatively, if the delete behavior is used, the key corresponding to any of the held locks is simply deleted.
    13. This can be used to create ephemeral entries that are automatically deleted by Consul.
    14. By default, the gossip based failure detector is used as the associated health check.
    15. This failure detector allows Consul to detect when a node that is holding a lock has failed and to automatically release the lock.

    Distributed locks with sessions

    A distributed semaphore can be useful when you want to coordinate many services, while restricting access to certain resources.

    STEP1: Create Session

    Json Payload:

    {
    "LockDelay": "15s",
    "Name": "my-service-lock",
    "Node": "xyz.com",
    "Behavior": "delete",
    "TTL": "30s"
}

    Request:

    curl -d @payload.json -X PUT http://xyz.com:8500/v1/session/create

{"ID":"7778f276-4de9-64ee-779c-fb63191ac86d"}

    STEP2: List sessions

    curl http://xyz.com:8500/v1/session/list

    STEP3: Get session info

    curl http://xyz.com:8500/v1/session/info/[session ID]

[{"ID":"3f0784ad-a86c-2c8b-0bd3-44582fff82a8","Name":"leader-lock","Node":"xyz.com","Checks":["serfHealth"],"LockDelay":15000000000,"Behavior":"delete","TTL":"30s","CreateIndex":921,"ModifyIndex":921}]

    STEP4: Delete session

    curl -X PUT http://xyz.com:8500/v1/session/destroy/[session ID]

true

    STEP5: Renew session

    curl -X PUT http://xyz.com:8500/v1/session/renew/[session ID]

    STEP6: Lock the KV entry

    curl -X PUT http://xyz.com:8500/v1/kv/rtm/leader/3ea7f492-cf3a-64e4-1b06-0a258ab95aa9?acquire=3ea7f492-cf3a-64e4-1b06-0a258ab95aa9
