    Blocking Queries

    how do Consul watches actually work internally?

    1. Many endpoints such as node, health, and scatalog support blocking queries.
    2. whenever a watch-loop of type 'service' for target service foobar starts, an API call of /v1/health/service/foobar?index=0 is made the first time.
    3. As a part of this API response, the Consul server will return a header called X-Consul-Index.
    4. This value is represented in internal raft log data structure by a field named modifyIndex.
    5. This header indicates the value of raft log index at which the last write to that particular service was successfully applied and gossiped.
    6. This is the current version of the requested resource from Consul catalog.
    7. With this in mind, let’s say in the case of our service foobar, Consul returned X-Consul-Index header value as x1 the first time.
    8. Consul SDKs store this value with them, and the next time make an API call of /v1/health/service/foobar?index=x1 in watch loop to detect any change in the health of service foobar.
    9. The nature of these blocking queries is such that they don't return a response until, for that particular endpoint, a modifyIndex greater than x1 is present or blocking query timeout is reached.
    10. If it times out, then Consul returns the response with the same value of index=x1 in X-Consul-Index header.
    11. Whenever a response on such blocking queries is received, Consul SDKs watch implementations check for a previously known value (in this case x1) against the new value returned in the X-Consul-Index header.
    12. If the value returned is greater than the last known value, they assume that something has changed in that endpoint and call that particular watch’s handler function.
    13. An important thing to note here is, they don’t look at the response of the API at all to decide if the Watch’s handler function should be triggered.
    14. It’s only based on consecutive X-Consul-Index header values.
    15. So even if two consecutive calls to /v1/health/service/foobar returned the same response, they are assumed to be different if the value of X-Consul-Index header in the second response is greater than the first one.

    Filtering

    1. Curl can also encode the query with the --data-urlencode parameter.
    2. When using the --data-urlencode parameter the default method is POST so the -G parameter is needed to set the request method to GET.
    3. Filtering will be executed on the Consul server before data is returned, reducing the network load.
    4. To pass a filter expression to Consul, with a data query, use the filter parameter. 
      curl -G <path> --data-urlencode 'filter=<filter expression>'

    Logical or :

    curl -G <path> --data-urlencode 'filter=<Expression 1> or <Expression 2>'

    Logical and :

    curl -G <path> --data-urlencode 'filter=<Expression 1> and <Expression 2>'

    Logical not :

    curl -G <path> --data-urlencode 'filter= not <Expression>'

    Grouping:

    curl -G <path> --data-urlencode 'filter=( <Expression 1> )'

    Matching:

    curl -G <path> --data-urlencode 'filter=<Matching Expression 1>'
