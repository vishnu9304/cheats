    K8s

    What is K8's?

    With containers, you can run a variety of software components across a cluster of generic servers.
    This can help to ensure availability and make it easier to scale resources.

    Oh! wait...! I have few questions.

    1. How can I ensure multiple instances of a piece of software are spread across multiple servers for high availability?
    2. How can I deploy new code changes and roll them out across the entire cluster?
    3. How can I create new containers to scale up to handle additional load?

    K8's (Orchestration Tool) is the answer for all the above questions.

    K8's Architecture

    List of components:

    1. docker - The container runtime to execute the containers on each nodes.
    2. kubeadm - Tool that simplifies the process of setting up the cluster.
    3. kubelet - Agent that manages the process of running the containers on each nodes.
    4. kubectl - Tool to interact with the cluster once it is up.
    5. Control Plane (only on k8's master) - Series of services that forms the k8's master.

    Note: The k8's control plane services will be running on containers under k8's master.

    Bootstrapping
    1. sudo setenforce 0
    2. sudo vi /etc/selinux/config
      Change the line that says SELINUX=enforcing to SELINUX=permissive and save the file.
    3. sudo systemctl enable kubelet
    4. sudo systemctl start kubelet
    5. sudo vi /etc/sysctl.d/k8s.conf
      net.bridge.bridge-nf-call-ip6tables = 1 net.bridge.bridge-nf-call-iptables = 1
    6. sudo sysctl --system
    7. sudo swapoff -a
    8. sudo kubeadm init --pod-network-cidr=10.244.0.0/16

    9. You will get a below message.

      To start using your cluster, you need to run the following as a regular user:
      mkdir -p $HOME/.kube
      sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
      sudo chown $(id -u):$(id -g) $HOME/.kube/config

      You should now deploy a pod network to the cluster.

      Run "kubectl apply -f [podnetwork].yaml" with one of the options listed at:
      https://kubernetes.io/docs/concepts/cluster-administration/addons/

      Then you can join any number of worker nodes by running the following on each as root:
      kubeadm join 10.82.219.85:6443 --token token --discovery-token-ca-cert-hash sha256:HASH

    10. kubectl apply -f https://raw.githubusercontent.com/coreos/flannel/bc79dd1505b0c8681ece4de4c0d86c5cd2643275/Documentation/kube-flannel.yml

    11. kubectl get nodes

    How to identify which runtime the docker is using? 

    sudo docker info | grep -i runtime
Runtimes: runc
Default Runtime: runc

    How to add another runtime?

    docker daemon --add-runtime "<runtime-name>=<runtime-path>"

Example:
docker daemon --add-runtime "oci=/usr/local/sbin/runc"

    What is containerd?

    1. containerd includes a daemon exposing gRPC API over a local UNIX socket.
    2. The API is a low-level one designed for higher layers to wrap and extend.
    3. Containerd uses RunC to run containers according to the OCI specification.

    Runc subsystem of containerd

    1. /usr/bin/docker-runc runc (OCI runtime) canbe seen as component of containerd.
    2. runc is a command line client for running applications packaged according to the OCI format and is a compliant implementation of the OCI spec.
    3. Containers are configured using bundles. A bundle for a container is a directory that includes a specification file named "config.json" and a root filesystem.
    4. The root filesystem contains the contents of the container.

    5. Assuming you have an OCI bundle you can execute the container:

      cd /mycontainer
      runc run mycontainerid

    containerd-shim
    It allows the runtimes, i.e. runc,to exit after it starts the container. This way we don't have to have the long running runtime processes for containers.
