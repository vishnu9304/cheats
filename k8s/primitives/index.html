    K8s Basics

    1. Pods are the smallest and most basic building block of the kubernetes model.
    2. A pod consist of one or more containers, storage resources and unique ip addresses in the k8's cluster network.
    3. In order to run containers, kubernetes schedules pods to run on servers in the cluster.
    4. When a pod is scheduled, the server will run the containers that are part of that pod.
    5. All objects in k8s will be having yaml definition.

    Example: nginx.yaml

    apiVersion: v1
kind: Pod
metadata:
    name: nginx
spec:
    containers:
        - name: nginx
          image: nginx

    Note: While working with k8s, every object will be in a namespace.

    Commands:

    kubectl get pods --all-namespaces -o wide - To get pods from all the namespaces.
    kubectl create -f [yaml definition file] - To create k8s objects.
    kubectl get pods - To get list of running pods. If no "namspace" is specified "default" namespace will be considered.
    kubectl describe pod [podname] - To get bunch of information about a particulat pod. kubectl get pods -n kube-system - To list pods under "kube-system" namespace.
    kubectl delete pod [podname] - To delete a pod.

    Namespaces:

    1. Multiple virtual clusters are backed by the same physical cluster.
    2. Generally for large deployments.
    3. Provide scope for names.
    4. Easy way to divide cluster resources.
    5. Allows multiple team of users.
    6. Allows for resource quotas.
    7. Special "kube-system" namespace, used to differentiate pods from user pods.

    Note: we can use lables to distinguish between two slightly different objects (Namespaces not required here).

    Core Concepts:

    1. Inter cluster communications happens with a call to API server.
    2. Every single k8's system components communicate with API server and they do not talk to each other directly.
    3. API server is the only one that communicates with the "etcd" data store.

    The kubectl command line utility can be used to create,update,delete and modify the api objects.

    Note: The api server exposes a resource called as component status which will show all the components of the control plane and status of each.

    kubectl get componentstatus
